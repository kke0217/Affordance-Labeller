# 실무 업무 할당

## 역할 정의
- **A 담당**: UI / 시각화
- **B 담당**: 데이터 구조 / 저장 / validation
- **C 담당**: 어노테이션 기준 정의 / 샘플 작성 / 검수 기준
- **D 담당(선택)**: 자동화 PoC 조사 및 후순위 실험

---

## 주차별 핵심 책임

### 1주차
#### A 담당
- Viser 환경 구성
- 입력 뷰 구현
- mug asset 표시

#### B 담당
- JSON skeleton 설계
- save / reload 구조 구현
- canonical frame 저장 구조 정의

#### C 담당
- YCB mug 기준 part 정의
- canonical frame 규칙 문서화
- 라벨링 기준 초안 작성

#### D 담당(선택)
- 자동화 관련 선행기술 정리만 수행
- 본개발 착수 금지

---

### 2주차
#### A 담당
- affordance 편집 UI 구현
- part 표시 및 수정 UI 구현
- 색상 오버레이 반영

#### B 담당
- affordance / part / tag 저장 구조 확정
- 재로드 동기화 처리

#### C 담당
- class / tag 기준안 확정
- 샘플 라벨 1개 작성
- 라벨링 기준 모호점 정리

#### D 담당(선택)
- 자동 초안 제안 삽입 지점 조사

---

### 3주차
#### A 담당
- contact mask 편집 UI 구현
- candidate pose 리스트 / 선택 UI 구현
- pose 좌표축 시각화 구현

#### B 담당
- mask / pose 저장 구조 구현
- affordance-mask-pose 연결 관계 저장
- validation 초안 작성

#### C 담당
- mask type 정의

- 샘플 1~2개 추가 작성

#### D 담당(선택)
- 간단 자동 제안 방식 검토
- 단, 메인 기능 완료 전 병합 금지

### 4주차

#### B 담당

- 작은 PoC가 가능할 경우만 연결
- 실패 시 즉시 제거 가능해야 함

## 업무 분배 원칙
- 새 기능 제안은 주간 게이트 통과 후 검토
- 저장 구조 변경은 B 담당 승인 후 반영
- class / tag / pose 명명 규칙은 C 담당 기준안 우선
- A 담당은 UI 과설계 금지
- D 담당 결과는 메인 기능을 흔들지 않는 선에서만 반영

## 완료 판정 기준
- A 담당: 사용자가 실제로 클릭/편집할 수 있는가
- B 담당: 저장 후 재로드 시 데이터가 살아있는가
- C 담당: 라벨 기준이 문서로 설명 가능한가
- D 담당: 붙였다 뗄 수 있는 작은 실험인가
